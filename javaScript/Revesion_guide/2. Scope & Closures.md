## **1. What is Scope in JavaScript?**

‚úî **Scope** defines the **accessibility of variables, functions, and objects** in different parts of the code.

‚úî JavaScript has **three types of scope:**

| **Scope Type**        | **Description**                                                                | **Example**                   |
| --------------------------- | ------------------------------------------------------------------------------------ | ----------------------------------- |
| **Global Scope**      | Accessible anywhere in the program.                                                  | `let x = 10;`                     |
| **Function Scope**    | Variables declared inside a function are accessible only inside that function.       | `function test() { let y = 20; }` |
| **Block Scope (ES6)** | Variables declared with `let`and `const`inside `{}`are not accessible outside. | `{ let z = 30; }`                 |

---

## **2. Global Scope Example**

üîπ Variables declared outside functions are **global** and accessible anywhere.

```js
let globalVar = "I am global"; 

function showGlobal() {
    console.log(globalVar); // ‚úÖ Accessible
}
showGlobal();
console.log(globalVar); // ‚úÖ Accessible
```

---

## **3. Function Scope Example**

üîπ Variables declared inside a function **cannot be accessed outside** the function.

```js
function showScope() {
    let functionVar = "I am inside a function";
    console.log(functionVar); // ‚úÖ Accessible
}
showScope();
console.log(functionVar); // ‚ùå Uncaught ReferenceError
```

---

## **4. Block Scope Example (ES6 - `let` & `const`)**

üîπ **`let` and `const`** have block scope, while  **`var` does not** .

```js
{
    let blockVar = "I am block-scoped";
    const blockConst = "I am also block-scoped";
    var notBlockScoped = "I am NOT block-scoped";
}

console.log(notBlockScoped); // ‚úÖ Accessible
console.log(blockVar); // ‚ùå ReferenceError
console.log(blockConst); // ‚ùå ReferenceError
```

---

## **5. Lexical Scope**

‚úî **Lexical Scope** means an **inner function** can access variables from  **outer functions** .

‚úî **Child functions inherit scope from parent functions.**

```js
function outer() {
    let outerVar = "I am from outer function";

    function inner() {
        console.log(outerVar); // ‚úÖ Accessible
    }

    inner();
}
outer();
```

---

## **6. Closures in JavaScript**

‚úî A **closure** is a function that  **remembers variables from its outer scope** , even after the outer function has finished executing.

‚úî Closures help **preserve state** and are useful in **encapsulation** and  **function factories** .

### **Example 1: Basic Closure**

```js
function outerFunction() {
    let count = 0; // Private variable

    return function innerFunction() {
        count++; 
        console.log(`Count: ${count}`);
    };
}

const counter = outerFunction();
counter(); // Count: 1
counter(); // Count: 2
counter(); // Count: 3
```

üîπ The **inner function retains access** to `count`, even though `outerFunction` has finished execution.

### **7. Use Cases of Closures in Interviews**

‚úÖ **1Ô∏è‚É£ Data Privacy / Encapsulation**

‚úî Closures **simulate private variables** in JavaScript.

```js
function createBankAccount(initialBalance) {
    let balance = initialBalance;

    return {
        deposit(amount) {
            balance += amount;
            console.log(`Deposited: ${amount}, New Balance: ${balance}`);
        },
        withdraw(amount) {
            if (amount > balance) {
                console.log("Insufficient Funds");
            } else {
                balance -= amount;
                console.log(`Withdrawn: ${amount}, Remaining Balance: ${balance}`);
            }
        }
    };
}

const myAccount = createBankAccount(1000);
myAccount.deposit(500);  // Deposited: 500, New Balance: 1500
myAccount.withdraw(200); // Withdrawn: 200, Remaining Balance: 1300
console.log(myAccount.balance); // ‚ùå Undefined (private variable)
```

‚úÖ **2Ô∏è‚É£ Function Factories (Dynamic Functions)**

‚úî Closures  **generate multiple functions with preserved states** .

```js
function createMultiplier(multiplier) {
    return function (num) {
        return num * multiplier;
    };
}

const double = createMultiplier(2);
const triple = createMultiplier(3);

console.log(double(4)); // 8
console.log(triple(4)); // 12
```

‚úÖ **3Ô∏è‚É£ Event Listeners (Retaining State in Callbacks)**

‚úî Closures  **help retain data inside event handlers** .

```js

    let count = 0;
    document.getElementById("myButton").addEventListener("click", function () {
        count++;
        console.log(`Button clicked ${count} times`);
    });
}
clickCounter();
```

‚úÖ **4Ô∏è‚É£ setTimeout and Closures**

‚úî Closures allow `setTimeout` to remember variables after the function execution.

```js

    setTimeout(() => {
        console.log(message);
    }, delay);
}

delayedMessage("Hello after 3 seconds!", 3000);
```

### ‚úÖ **Q1: What is the Difference Between Lexical Scope and Closure?**

| Feature              | **Lexical Scope**                                                             | **Closure**                                                                       |
| -------------------- | ----------------------------------------------------------------------------------- | --------------------------------------------------------------------------------------- |
| **Definition** | Determines**where variables are accessible**based on where they are declared. | A function**"remembers"**variables from its**lexical scope**even after execution. |
| **Concept**    | **Scope chain**follows **nested function rules** .                      | Function**retains access**to outer function‚Äôs variables even after execution.    |
| **Access**     | Inner functions can access outer variables**only when called inside** .       | Inner function can access outer variables**even when called outside** .           |

---

### **1Ô∏è‚É£ Lexical Scope (Defined at Declaration, Not Execution)**

* **Lexical Scope** means that  **a function‚Äôs scope is determined by where it is written, not where it is executed** .

```js
function outer() {
  let outerVar = "I am from outer!";
  
  function inner() {
    console.log(outerVar); // ‚úÖ Accessible due to lexical scope
  }
  
  inner();
}

outer(); // "I am from outer!"
```

‚úÖ **Inner function `inner()` can access `outerVar` because it is declared inside `outer()`'s scope.**

---

### **2Ô∏è‚É£ Closure (Function "Remembers" Its Lexical Scope)**

* A **closure** happens when an inner function is **returned** and retains access to its  **outer function‚Äôs variables** , even after the outer function finishes executing.

```js
function outer() {
  let count = 0; // Lexical scope variable

  return function inner() {
    count++;
    console.log(`Count: ${count}`);
  };
}

const counter = outer(); // `outer()` returns `inner()`, keeping `count` alive
counter(); // Count: 1
counter(); // Count: 2
counter(); // Count: 3
```

‚úÖ **Even though `outer()` has finished execution, `inner()` still has access to `count`!**

‚úÖ This is a  **closure** , as `inner()` "remembers" `count` from its  **lexical scope** .

---

### **3Ô∏è‚É£ Key Differences & Interview Tip**

‚úî **Lexical Scope** ‚Üí Defines which variables a function  **can access** .

‚úî **Closure** ‚Üí **Keeps** those variables alive even after the function executes.

‚úî **Example Answer:** *"Lexical scope is where variables are accessible based on function nesting, while a closure allows a function to remember and use variables from its lexical scope even after execution."* üöÄ



### ‚úÖ **Q2: How Do Closures Help in JavaScript?**

Closures are a powerful feature in JavaScript that allow functions to  **retain access to variables from their lexical scope** , even after the outer function has finished executing.

---

### **1Ô∏è‚É£ Benefits of Closures in JavaScript**

‚úî **Data Encapsulation (Private Variables)** ‚Äì Helps create private variables that cannot be accessed directly.

‚úî **State Persistence** ‚Äì Retains values between function calls.

‚úî **Memoization & Caching** ‚Äì Stores previously computed values to improve performance.

‚úî **Event Listeners & Callbacks** ‚Äì Allows handling events effectively.

‚úî **Function Factories** ‚Äì Returns functions customized with different initial values.

### **2Ô∏è‚É£ Use Cases of Closures**

#### üîπ **1. Data Encapsulation (Private Variables)**

Closures allow us to create **private variables** that cannot be accessed from outside.

```js
function createCounter() {
  let count = 0; // Private variable

  return {
    increment: function () {
      count++;
      console.log(`Count: ${count}`);
    },
    decrement: function () {
      count--;
      console.log(`Count: ${count}`);
    },
  };
}

const counter = createCounter();
counter.increment(); // Count: 1
counter.increment(); // Count: 2
counter.decrement(); // Count: 1
// console.log(counter.count); ‚ùå Undefined (cannot access private variable)
```

‚úÖ **Why?** `count` remains accessible  **only inside the returned functions** .

---

#### üîπ **2. State Persistence (Remembering Values)**

Closures **retain state** even after the outer function finishes execution.

```js
function greeting(name) {
  return function () {
    console.log(`Hello, ${name}!`);
  };
}

const greetJohn = greeting("John");
greetJohn(); // ‚úÖ "Hello, John!"
greetJohn(); // ‚úÖ Still remembers "John"
```

‚úÖ **Why?** `name` persists even after `greeting()` has executed.

---

#### üîπ **3. Memoization (Optimizing Expensive Calculations)**

Closures help in caching results of expensive function calls.

```js
function memoizedFactorial() {
  let cache = {};

  return function factorial(n) {
    if (n in cache) {
      console.log("Fetching from cache");
      return cache[n];
    }
    console.log("Calculating result");
    let result = n === 0 ? 1 : n * factorial(n - 1);
    cache[n] = result;
    return result;
  };
}

const fact = memoizedFactorial();
console.log(fact(5)); // Calculating result -> 120
console.log(fact(5)); // Fetching from cache -> 120
```

‚úÖ **Why?** The `cache` variable **remains in memory** due to closure, improving performance.

---

#### üîπ **4. Function Factories (Generating Functions Dynamically)**

Closures allow us to  **create functions dynamically** .

```js
function multiplier(factor) {
  return function (num) {
    return num * factor;
  };
}

const double = multiplier(2);
console.log(double(5)); // ‚úÖ 10

const triple = multiplier(3);
console.log(triple(5)); // ‚úÖ 15
```

‚úÖ **Why?** Each function retains its own `factor` value.

### ‚úÖ **Q3: Can Closures Be Used for Private Variables?**

‚úî **Yes! Closures are an excellent way to implement private variables in JavaScript.**

‚úî Private variables **cannot be accessed directly** from outside the function but  **can be manipulated through inner functions** .

---

### **1Ô∏è‚É£ Why Use Closures for Private Variables?**

üîπ **Encapsulation:** Hides variables from global scope.

üîπ **Prevents Unwanted Modification:** Users can't directly change sensitive data.

üîπ **Controlled Access:** Provides getters & setters for safe updates.

---

### **2Ô∏è‚É£ Example: Creating Private Variables Using Closures**

```js
function createBankAccount(initialBalance) {
  let balance = initialBalance; // Private variable

  return {
    deposit: function (amount) {
      balance += amount;
      console.log(`Deposited: ${amount}, New Balance: ${balance}`);
    },
    withdraw: function (amount) {
      if (amount > balance) {
        console.log("Insufficient funds!");
      } else {
        balance -= amount;
        console.log(`Withdrawn: ${amount}, New Balance: ${balance}`);
      }
    },
    getBalance: function () {
      return balance;
    },
  };
}

const myAccount = createBankAccount(1000);
myAccount.deposit(500); // ‚úÖ Deposited: 500, New Balance: 1500
myAccount.withdraw(200); // ‚úÖ Withdrawn: 200, New Balance: 1300
console.log(myAccount.getBalance()); // ‚úÖ 1300
console.log(myAccount.balance); // ‚ùå Undefined (Cannot access private variable)
```

‚úÖ **Why?** `balance` is private and can only be modified through methods inside the returned object.

---

### **3Ô∏è‚É£ Alternative: Using Closures for Counter with Private Variable**

### ‚úÖ **Q4: What Happens If We Create Multiple Closures?**

‚úî When we create multiple closures,  **each closure retains its own reference to the variables in its lexical scope** .

‚úî This means that every closure maintains **a separate copy** of the variables from the outer function.

---

### **1Ô∏è‚É£ Multiple Closures ‚Äì Independent State**

Each closure has **its own scope** and does not interfere with others.

```js
function createCounter() {
  let count = 0; // Private variable for each closure

  return function () {
    count++;
    console.log(`Count: ${count}`);
  };
}

const counter1 = createCounter();
const counter2 = createCounter();

counter1(); // ‚úÖ Count: 1
counter1(); // ‚úÖ Count: 2
counter2(); // ‚úÖ Count: 1 (Separate instance)
counter2(); // ‚úÖ Count: 2 (Still independent)
```

‚úÖ **Why?** `counter1` and `counter2` have  **separate memory for `count`** , so they do not affect each other.

---

### **2Ô∏è‚É£ Common Mistake ‚Äì Sharing the Same Scope**

If closures  **share the same reference** , they can  **overwrite each other's values** .

```js
function sharedCounter() {
  let count = 0;
  return {
    increment: () => console.log(++count),
    decrement: () => console.log(--count),
  };
}

const counterA = sharedCounter();
const counterB = counterA; // Both refer to the same closure

counterA.increment(); // ‚úÖ 1
counterA.increment(); // ‚úÖ 2
counterB.decrement(); // ‚ùå 1 (Affects counterA)
counterB.decrement(); // ‚ùå 0 (Still affecting counterA)
```

‚ùå **Issue:** `counterA` and `counterB` **share the same closure** instead of creating separate ones.

---

### **3Ô∏è‚É£ Closures Inside Loops (Common Pitfall)**

Closures inside loops can sometimes behave unexpectedly due to  **variable capturing** .

#### ‚ùå **Incorrect Closure (Without Block Scope)**

```js
function createTimers() {
  for (var i = 1; i <= 3; i++) {
    setTimeout(() => console.log(`Timer ${i}`), i * 1000);
  }
}
createTimers();

// Output after 1s, 2s, 3s: ‚ùå
// Timer 4
// Timer 4
// Timer 4
```

**Why?** The `var` keyword creates a single shared `i`, which is **4** when the timeout executes.

---

#### ‚úÖ **Correct Closure (Using `let` for Block Scope)**

```js
function createTimers() {
  for (let i = 1; i <= 3; i++) {
    setTimeout(() => console.log(`Timer ${i}`), i * 1000);
  }
}
createTimers();

// Output after 1s, 2s, 3s: ‚úÖ
// Timer 1
// Timer 2
// Timer 3
```

**Why?** `let` creates a  **new `i` for each iteration** , preserving the correct value.

---

### **4Ô∏è‚É£ Interview Tip**

‚úî **Explain in One Line:** *"Each closure maintains its own reference to the lexical scope, but if a closure captures a shared variable, all instances may refer to the last updated value."*

‚úî **Use an Example:** Mention  **counters, loops, or separate function instances** . üöÄ



### ‚úÖ **Q5: How Does JavaScript Manage Memory with Closures?**

‚úî JavaScript uses **Garbage Collection (GC)** to free up memory, but closures  **retain references to their lexical scope** , which can sometimes cause **memory leaks** if not handled properly.

---

### **1Ô∏è‚É£ Closures & Memory Retention**

üîπ When a function creates a closure, it **retains access** to its outer variables, even after the outer function has executed.

üîπ This prevents the **JavaScript garbage collector** from reclaiming memory because the variables are still referenced.

#### ‚úÖ **Example: Closure Holding Memory**

```js
function createClosure() {
  let largeData = new Array(1000000).fill("üöÄ"); // Large data structure

  return function () {
    console.log(largeData[0]); // Retains reference
  };
}

const closureFn = createClosure(); // The largeData array is still in memory
```

‚úÖ **Issue:** The `largeData` array is **not garbage collected** because `closureFn` still holds a reference to it.

---

### **2Ô∏è‚É£ Memory Leaks with Closures**

Closures can cause **memory leaks** when references to variables  **are unintentionally kept in memory** .

#### ‚ùå **Memory Leak Example**

```js
function createLeak() {
  let bigArray = new Array(1000000).fill("üî•");

  return function () {
    console.log("Using bigArray");
  };
}

const leakyFunction = createLeak();
// If leakyFunction is not used again, bigArray should be garbage collected,
// but it stays in memory because it's still referenced.
```

üö® **Problem:** The `bigArray` is **never freed** because `leakyFunction` still has a reference.

---

### **3Ô∏è‚É£ Preventing Memory Leaks in Closures**

‚úî **Solution 1: Explicitly Remove References**

```js
let closureFn = createClosure();
closureFn = null; // ‚úÖ Allows garbage collection of the retained data
```

‚úÖ This removes the reference, allowing garbage collection.

---

‚úî **Solution 2: Use WeakMap or WeakSet (For Object References)**

WeakMap **does not prevent garbage collection** of unused objects.

```js
const weakCache = new WeakMap();

function storeData(key, value) {
  weakCache.set(key, value);
}

let obj = {};
storeData(obj, new Array(1000000).fill("üß†"));
obj = null; // ‚úÖ Memory is freed when there are no more references to obj
```

‚úÖ **Why?** When `obj` is set to `null`, the WeakMap entry is  **automatically garbage collected** .

---

‚úî **Solution 3: Avoid Unused Closures**

Closures should  **only store necessary data** .

‚ùå **Bad Example (Holds unnecessary data)**

```js
function badClosure() {
  let unusedData = new Array(1000000).fill("‚ùå");

  return () => console.log("Hello");
}
```

‚úÖ **Good Example (Releases memory)**

```js
function goodClosure() {
  let message = "Hello"; // Only keeps required data

  return () => console.log(message);
}
```

---
